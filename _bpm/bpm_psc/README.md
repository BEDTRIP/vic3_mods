## Итог по конфликтам PSC ↔ BPM (порядок загрузки: **PSC → BPM**)

Сгенерировал отчёт в `bpm_psc/conflicts_psc_vs_bpm_report.md`.

### Жёсткие конфликты (одинаковый путь файла)

- **Нет**: пересечений по одинаковым относительным путям файлов **0** (в т.ч. case-insensitive тоже 0). Значит BPM **ничего не перезатирает “файлом целиком”** из PSC.

### Пересечения в `common/*` (одинаковый top-level ключ)

Это **единственные** найденные пересечения:

- *`common/history/global`: `GLOBAL**`
  - PSC: `common/history/global/PSC_global.txt` (триггерит `set_construction_start`)
  - BPM: `common/history/global/00_bpm_global.txt`, `zzzzzz_bpm_global_last.txt` (свои эффекты)
  - По виду это **аддитивные блоки** (без `REPLACE:`), т.е. должны сосуществовать.
- *`common/on_actions`: `on_acquired_technology**`
  - PSC: `common/on_actions/PSC_on_actions.txt` (хуки для обновления PSC-конверсии)
  - BPM: `common/on_actions/BPM_code_on_actions.txt` (свои tech-реакции)
  - Тоже **без `REPLACE:**`, выглядит как **обычное добавление on_action’ов**.

### Остальное (проверено дополнительно)

- `localization` *дубликатов ключей**: 0
- *`events` дубликатов `id = ...`**: 0
- В BPM **нет** определений тех ванильных сущностей, которые PSC явно заменяет (строительный сектор/его PM`bg_construction`/алерты/defines) — прямых геймплей-конфликтов там не обнаружилось.

### Что может потребовать “страховочного” компача

Если вдруг окажется, что у Vic3 в вашей версии `GLOBAL` *или* `on_acquired_technology` *не мерджатся** (а перезаписываются последним определением), то PSC может потерять свои хуки. В таком случае в `bpm_psc` имеет смысл сделать маленький “страховочный” файл, который **явно собирает оба блока** (обычно через `REPLACE:`) — но это уже следующий шаг при сборке компача.
